# (S)AGE Protocol — Gap Analysis Against Optimus SPEC v0.7.0

> **Research question:** Does the (S)AGE institutional memory protocol address gaps in Optimus's agent knowledge accumulation, or introduce patterns worth adopting?
> **Date:** 2026-03-08
> **Spec sections examined:** §3 (Task Graph), §4 (Agent Runtime — context management), §5 (Guardrail Enforcement — content sanitization), §8 (Audit and Observability — threat memory, behavioral drift), §11 (Failure Modes — agent replacement), §13 (AutoBot Constitution), §19 (Strategy Evaluation Protocol), §20 (Deferred Items)
> **Source:** [l33tdawg/sage](https://github.com/l33tdawg/sage) — 4 papers (Zenodo DOIs), Go implementation, Python SDK

---

## Executive Summary

(S)AGE ("Sovereign Agent Governed Experience") is a Byzantine fault-tolerant institutional memory layer for multi-agent systems. It solves a specific problem Optimus currently defers: **how do agents learn from accumulated operational experience, not just from their prompts and immediate task context?**

The core thesis — agents with access to consensus-validated institutional memory outperform agents relying solely on prompt engineering — is empirically validated in their Paper 2 (40% lower calibration error, Cohen's d = -0.824) and Paper 4 (statistically significant learning trend, Spearman ρ = 0.716, p = 0.020). Paper 3 goes further: 3-line prompts with institutional memory outperformed expert-crafted 120-line domain-specific prompts.

**Bottom line for Optimus:** (S)AGE confirms that our `state_transitions` + `strategic_decisions` + `threat_memory` tables are the right data substrate — but we have no mechanism to distill that data into queryable institutional knowledge that improves agent performance over time. This is the gap. We don't need their BFT consensus layer (we have Postgres + hash chains), but their insight about structured knowledge accumulation is directly applicable to Phase 2+.

**Paper 4 operational warning:** Knowledge accumulation alone is insufficient. Paper 4's two highest-impact findings are both about *knowledge plumbing*, not knowledge content: (1) agents must be explicitly routed to the right knowledge domains — two wiring fixes that connected agents to existing knowledge unlocked the entire learning trajectory; (2) domain taxonomy pollution (misclassified entries) caused catastrophic regression even though the knowledge base contained all the right information. For Optimus, this means our Phase 1 context loading architecture (§4 step 5) needs instrumentation to detect routing gaps and quality filtering to prevent noise from drowning signal — both low-cost additions to existing infrastructure.

**Recommendation:** DEFER adoption to Phase 2. For Phase 1, instrument the data that would feed an institutional memory system (we're already doing this), PLUS add context coverage logging to detect knowledge routing gaps (new from Paper 4 analysis) and enforce tier-based quality filtering in prior work search (new from Paper 4 analysis). In Phase 2, build a lightweight knowledge distillation layer over `state_transitions` and `strategic_decisions` that surfaces patterns to agents at context-loading time (§4 step 5). Add as SPEC-ADDENDUM entry.

## Validation (Spec Is Correct)

These (S)AGE findings confirm Optimus's current approach:

**1. Append-only audit trails are foundational (§3, §8).** (S)AGE's entire memory lifecycle (proposed → committed → challenged → deprecated) relies on append-only state with cryptographic integrity. Our `state_transitions` and `threat_memory` tables with hash chains and `REVOKE TRUNCATE/DROP/ALTER` are architecturally aligned. Their approach uses BadgerDB for on-chain hash state + PostgreSQL for full content — similar to our split between hash chains (tamper detection) and full records (operational data). Paper 1 §3 specifies parent revision pointers forming an append-only revision chain with SHA-256 hash commitments — structurally identical to our `state_transitions.prev_hash` design.

**2. Content sanitization before context loading is critical (§5).** (S)AGE's threat model (Paper 1 §5) explicitly names "adversarial prompt injection via memory" as a CRITICAL vector — embedding adversarial prompts within memory objects that manipulate consuming agents on retrieval. Their mitigation: "Memory content is rendered as data, never executed as instructions; consuming agents receive memories in sandboxed context." This directly validates our §5 content sanitization pipeline (step 4f) and our P2 principle — infrastructure enforcement, not prompt-level trust. Their PoE validation quorum provides an additional layer our Q1-Q4 tiers don't have: knowledge must pass a weighted vote *before* entering the queryable pool. Worth considering for Phase 2 when we build the knowledge distillation layer (G1).

**3. Structured records beat prose (§19).** Paper 2 demonstrates that structured, validated memory records improve agent performance more than detailed prompt engineering. The Paper 2 prompt ablation study is the key evidence: an 18-line "onboarding" prompt (job scope, scoring rubric, output format — zero domain knowledge) with access to curated institutional memory achieved perfect calibration (gap = 0.0), outperforming an expert-crafted 120-line domain-specific prompt whose best result across 20 runs was gap = 0.10. This directly validates our `strategic_decisions` schema design and the broader principle that agents referencing structured decision records produce better outcomes than agents given more elaborate instructions.

**4. Hierarchical agent organization with RBAC works (§2, §5).** Paper 3's CipherForge Labs uses 11 agents across 5 departments with clearance levels (0-4: None → Read → Read+Write → Read+Write+Validate → Admin). Their department-scoped RBAC maps almost exactly to our role-level guardrails table (§5): the Red Team at clearance 2 can read across departments, the CEO at clearance 3 has full organizational access. This validates our `data_classification_clearance` field in agent configs and confirms that hierarchical organization with information barriers is the right pattern — not flat agent meshes where everyone sees everything.

**5. Seed knowledge bootstrapping validates `agents.md` + config approach (§4).** Paper 3 bootstraps all 11 agents from 8 seed knowledge entries (~2,500 words) covering company handbook, scoring rubric, difficulty definitions, pipeline workflow, department responsibilities, and domain-specific baselines. This is functionally equivalent to our `agents.md` + compiled JSON config approach (§4): human-authored organizational knowledge that gets loaded as foundational context. Their finding that 11/11 agents correctly understood their role, department, pipeline position, and executive hierarchy from seed knowledge alone validates our context-loading architecture — particularly the priority ordering where agent identity + guardrails load first (step 4a-e).

**6. Echo chamber is a real failure mode requiring active mitigation.** Paper 2 identifies that without diversity controls, institutional memory converges to a single dominant pattern — all 20 sequential runs generated the same vulnerability type (Padding Oracle), and each observation reinforced the pattern. They call this analogous to recommendation system filter bubbles. Two mitigations were validated: knowledge seeding at initialization (7 of 10 curated seeds accepted by BFT quorum) and novelty-aware summarization (the summarizer receives existing knowledge and derives only net-new insights). This validates our §19 perspective divergence metric and the §14 Phase 3 social physics exploration ratio circuit breaker, and adds a concrete mitigation pattern (novelty-aware summarization) we should adopt when building the knowledge distillation pipeline.

**7. Declarative routing tables validate our state machine approach (§3).** Paper 3's pipeline uses a declarative routing table (20+ states with explicit transitions for happy paths, failure/repair loops, hardening cycles, and terminal states) rather than a monolithic orchestrator. Their `resolve_outcome()` function inspects context after each agent executes — structurally parallel to our `transition_state()` function with `valid_transitions` table enforcement. This confirms that event-driven, state-machine-governed orchestration is the right pattern at our agent count (5-15).

**8. Confidence decay addresses knowledge staleness (Paper 1 §4.4).** (S)AGE's memory confidence decays exponentially over time with domain-specific λ rates (λ=0.001 for stable domains like cryptography, λ=0.01 for fast-moving domains like vulnerability intelligence). Memories below a minimum confidence threshold get flagged for revalidation or archival. This is a pattern we should consider for Phase 2 institutional knowledge — not all learned patterns remain valid as models, tools, and task types evolve.

**9. Identity via public key cryptography vs. JWT — different trust models.** (S)AGE uses Ed25519 keypairs for agent identity, with signatures on every memory submission providing cryptographic non-repudiation. Our JWT-scoped identity serves the same purpose with a critical advantage for governed systems: revocability (§9 step 5). Their model is better for federated/sovereign deployments where validators may be semi-trusted. Ours is better when the board controls all agents and the ability to instantly revoke identity is more valuable than cryptographic non-repudiation. Both approaches work; the choice depends on trust model.

**10. Regression-recovery cycles validate graduated trust escalation (§11, Paper 4).** Paper 4's RT difficulty trajectory is not monotonic: runs 15-16 regress from 2.5 to 1.8, then recover to 2.5 (run 17) and ultimately reach 3.0 (run 20). This non-linear learning curve validates our graduated trust escalation design — trust level resets when rejection rate exceeds 10% in any 7-day window (§11). A system that assumes monotonic improvement would miss regressions; a system with measurement-based trust gates catches them and requires re-earning autonomy. Paper 4 proves this happens in practice even when the underlying knowledge base is growing correctly.

**11. Knowledge quality dominates knowledge quantity (Paper 4 §5, domain cleanup).** Paper 4's most operationally significant finding: after domain tag pollution (44 misclassified entries drowning out 18 curated reports), the same designer agent with the same prompts and the same 18 valid reports regressed from RT 2.5 to RT 1.0. After cleanup (re-tagging noise to a separate domain), performance immediately recovered to RT 2.5. The only variable was retrieval precision — 5/5 relevant results vs. 1.45/5. This directly validates our Q1-Q4 data quality tiers (§4): the tiering ensures that high-provenance data (Q1: board-authored, Q2: reviewer-validated) is loaded before lower-quality data (Q3-Q4), and that lower tiers are capped as a percentage of context budget. The Paper 4 finding adds urgency to getting the tier boundaries right — a system that loads noise alongside signal doesn't degrade gracefully; it degrades catastrophically.

**12. Dual-sided feedback loops validate the Gateway + Reviewer architecture (§7, §5, Paper 4).** Paper 4 demonstrates that institutional memory accelerates improvement on both the defensive side (challenge designers learn what solvers break) and the offensive side (red team learns which shortcuts work). This dual-sided acceleration maps directly to our architecture: the Communication Gateway (§7) processes inbound signals that feed the task graph, while the Reviewer (§5) validates outputs that feed future context loading. The closed loop — Gateway inbound → task → Reviewer validation → knowledge → future agent context — is the same structure Paper 4 validated, just applied to business operations instead of CTF challenges.

**13. Innovation types evolve qualitatively, not just quantitatively (Paper 4).** Paper 4 categorizes the evolution of agent output across runs: baseline → targeted hardening → information hiding → architectural innovation → defense refinement → fundamental innovation → mathematical novelty. The agents didn't just get "better" — they developed qualitatively different strategies as knowledge accumulated. Run 14 independently discovered defense-in-depth (combining CBC and GCM) without any prompt instruction to do so. Run 17 modified the mathematical foundation of GHASH itself. This validates our Phase 2+ trajectory where agent capabilities expand through accumulated operational data (P5), not through prompt rewrites — but it also means our measurement infrastructure must track *types* of improvement, not just success rate metrics.

## Recommendations

**1. Add SPEC-ADDENDUM entry for institutional knowledge layer (Phase 2).** This is the primary gap. Draft a new §4.6 subsection: "Institutional Knowledge Distillation." Define the pipeline (completed tasks → pattern extraction → Reviewer validation → structured knowledge records → confidence decay → context loading), the storage schema, and the context-loading integration point. Gate activation on Phase 1 data from `v_routing_class_effectiveness` and `v_cost_per_task_type_trend`.

**2. Add `v_longitudinal_learning` analytical view to Phase 1 instrumentation.** This costs nothing to add now and produces the measurement data needed to justify G1. SQL is straightforward — success rate by task type by week, with Spearman correlation as the key metric.

**3. Add structured "lessons learned" to Reviewer workflow (Phase 1 prep).** Paper 3's closed feedback loop is the most architecturally significant finding for Optimus. When a Reviewer approves a task, capture a structured annotation: what approach worked and whether the pattern is generalizable. When a task fails and is retried successfully, capture the delta. This is a low-cost Phase 1 addition (a JSONB field on the `state_transitions` record for review transitions) that creates the raw material for Phase 2 knowledge distillation. **Board decision needed:** Should this be added to Phase 1 scope, or deferred?

**4. Adopt novelty-aware context loading as a Phase 2 optimization.** Paper 2's finding that all 20 runs generated the same vulnerability type due to homogeneous knowledge is directly relevant to our context loading. When multiple completed tasks say the same thing, loading all of them wastes tokens. Deduplicating near-similar context blocks before loading is a token optimization with direct cost impact (§15).

**5. Do NOT adopt (S)AGE's BFT consensus layer.** Optimus doesn't need Byzantine fault tolerance for knowledge validation — we operate a governed system where the Reviewer agent and hash chains provide sufficient integrity. BFT consensus adds CometBFT infrastructure complexity that violates P4 (boring infrastructure). Their 4-node BFT network solves a different trust model (permissioned validators who may be adversarial). In Optimus, the board controls all agents — there are no adversarial validators.

**6. Do NOT adopt Ed25519 agent identity.** JWTs with revocation (§9) are strictly more powerful for a governed system. Ed25519 provides cryptographic non-repudiation, but we get non-repudiation from the `state_transitions` audit trail with `config_hash` stamping.

**7. Do NOT adopt their "3-line prompt" pattern wholesale.** Paper 3's finding that minimal prompts + institutional memory outperform expert prompts is compelling, but it depends on having a mature institutional knowledge base. In Phase 1, we have zero accumulated knowledge — our agents need substantive prompts (via `agents.md`) to function. The correct trajectory: start with detailed `agents.md` definitions, build institutional knowledge through Phase 1 operations, then *measure* whether prompt complexity can be reduced in Phase 2 as the knowledge layer takes over. Consistent with P5 — measure before you trust.

**8. Monitor (S)AGE for Phase 3+ federation model.** Their Sovereign layer (organizations, departments, RBAC, cross-org federation with bilateral agreements, clearance gating, and on-chain enforcement) is relevant if Optimus products need to federate knowledge across customer deployments. The clearance-gated federation model is a well-designed pattern worth studying when multi-tenant agent identity (§20) becomes active.

**9. Consider confidence decay for institutional knowledge (Phase 2+).** Paper 1's domain-specific decay rates (λ=0.001 for stable domains, λ=0.01 for fast-moving ones) are a clean solution to knowledge staleness. When we build the institutional knowledge table, include a confidence field with configurable decay and corroboration bonuses.

**10. Add context coverage validation to task assignment (Phase 1, from Paper 4 G10).** Paper 4's knowledge routing failure is the highest-impact operational finding for Optimus. The fix is low-cost: when the Orchestrator assigns a task and constructs context, log which data sources were queried and which were available but not queried. Add this to `context_profile_json` (already specified in §3 for pathway instrumentation). This creates the measurement data to detect routing gaps before they cause silent performance degradation. No new schema needed — `context_profile_json` already exists.

**11. Enforce write-side quality controls on context pool contributions (Phase 1, from Paper 4 G11).** Paper 4's domain pollution finding maps directly to our context loading architecture. In Phase 1, implement a simple filter: when loading "relevant prior work" (§4 step 5), weight results by the contributing agent's tier. A Reviewer-validated output (Q2) should rank higher than a raw Executor output (Q3). This doesn't require a new table — it's a query-time filter using existing `state_transitions.agent_id` joined to `agent_configs.role`. Cost: near zero. Impact: prevents the noise-drowning-signal failure mode Paper 4 documented.

**12. Track innovation types in Reviewer annotations (Phase 2+, from Paper 4 G12).** Paper 4's innovation taxonomy (baseline → hardening → information hiding → architectural innovation → fundamental innovation) reveals that quantitative metrics alone miss the most important learning signal. When we build the lessons_learned annotation (Recommendation 3 / G7), include a `strategy_type` field: `established_pattern | variation | novel_approach`. This feeds the echo chamber detection metric (G5) with qualitative data.

### §4.6 Institutional Knowledge Layer (NEW)

> **Source:** (S)AGE research analysis, 2026-03-08
> **Spec section affected:** §4 (Agent Runtime — context management)
> **Change type:** NEW

#### Content

Completed task outputs in the `state_transitions` and `strategic_decisions` tables represent raw operational data. For agents to improve over time (longitudinal learning), this data must be distilled into queryable institutional knowledge — structured patterns that agents access at context-loading time (step 5).

**The core insight (from (S)AGE Papers 2-3):** An agent with minimal identity context ("who you are, where you work, what you're evaluated on") plus access to validated institutional knowledge outperforms an agent with detailed expert-crafted instructions but no institutional knowledge. Paper 3 demonstrates this at organizational scale: 11 agents, 5 departments, 3-line prompts each, producing 93.0/100 quality output on the first attempt from a cold start with only 8 seed knowledge entries. The implication for Optimus: as institutional knowledge accumulates through Phase 1 operations, the relative importance of detailed `agents.md` definitions may decrease — but this must be measured (P5), not assumed.

**Phase 1 deliverables (low cost, high signal):**
1. Add `v_longitudinal_learning` analytical view measuring per-task-type success rate trend over time.
2. **Board decision required:** Add `lessons_learned_json` JSONB field to `state_transitions` records where `to_state = 'completed'` AND the transition originated from review approval. The Reviewer annotates: what approach worked, whether the pattern is generalizable, and (for retry-after-failure cases) what changed between the failed and successful attempts. This captures the raw material for Phase 2 knowledge distillation at near-zero cost.
3. All other Phase 1 instrumentation (pathway views §8, decision records §19) already captures the data that feeds the knowledge pipeline.
4. **Add context source coverage to `context_profile_json` (Paper 4 G10).** When the orchestration layer loads context (step 4), record not just which blocks were loaded and their token counts, but also which data sources were *available but not queried*. Example: `{"queried_sources": ["parent_summary", "sibling_statuses", "prior_work_keyword"], "available_not_queried": ["prior_work_workstream_0042", "strategic_decisions_directive_007"], "total_context_tokens": 1140}`. This creates the measurement data to detect knowledge routing gaps — the Paper 4 equivalent of the designer not querying `red_team.verification`. Near-zero cost; extends existing `context_profile_json` field.
5. **Enforce tier-based filtering in prior work search (Paper 4 G11).** When loading "relevant prior work" (step 5), weight results by contributing agent's role tier: Reviewer-validated outputs (Q2) rank above raw Executor outputs (Q3). Implement as a query-time filter joining `state_transitions.agent_id` to `agent_configs.role`. This prevents the noise-drowning-signal failure mode Paper 4 documented (44 low-value entries drowning 18 high-value reports in semantic search). No new tables needed.

**Phase 2 activation gate:** `v_context_block_correlation` shows keyword search retrieving irrelevant context >20% of the time, AND the task graph contains >500 completed tasks, AND `v_longitudinal_learning` shows flat or declining trend (ρ < 0.1) for mature task types (indicating the system isn't learning from raw task history alone).

**Phase 2 implementation outline:**
- Knowledge distillation pipeline: utility agent processes completed task batches, extracts patterns, stores as structured records in `institutional_knowledge` table
- Reviewer validation: distilled knowledge scored for generalizability before entering the queryable pool (lighter-weight analog of (S)AGE's PoE consensus)
- Confidence decay: each knowledge record has a confidence score with domain-specific exponential decay rate (λ configurable per task type; stable patterns decay slowly, model-specific patterns decay quickly). Memories below threshold are archived, not deleted (append-only). Corroboration by subsequent successful tasks provides a stabilizing bonus.
- Novelty-aware loading: at distillation time, existing knowledge is provided to the summarizer to derive only net-new insights (prevents echo chamber formation). At context-loading time, near-duplicate blocks are collapsed before loading (token optimization).
- Diversity monitoring: track distinct pattern count per task type; alert if any task type converges to fewer than N distinct patterns in `institutional_knowledge` (echo chamber detection, per (S)AGE Paper 2)
- Context integration: add institutional knowledge as a Q2-tier context source at step 5, between "parent task summary" and "sibling task statuses"
- Embedding-based retrieval: add pgvector extension + local embedding model for semantic search over institutional knowledge (replaces keyword matching for this context source)

**Closed feedback loop (Paper 3 pattern):** When operational outcomes reveal actionable patterns (e.g., a Reviewer identifies that tasks of type X consistently fail when using approach Y but succeed with approach Z), the pattern is distilled, validated, and committed to the knowledge base. Future agents working on type X tasks receive this knowledge at context-loading time. This creates the autonomous improvement cycle (S)AGE demonstrated: create → execute → evaluate → learn → improve.

**Prompt complexity trajectory (Phase 2+ measurement):** Once institutional knowledge is actively loaded, measure whether `agents.md` complexity can be reduced without degrading task success rate. Paper 3 suggests that organizational identity ("who you are, where you work, what you're evaluated on") plus institutional knowledge may be sufficient for routine operational tasks, with detailed instructions reserved for novel task types. This is a P5-gated experiment, not an assumption.

#### Phase Activation
Phase 2, gated on measurement data from Phase 1 pathway views and `v_longitudinal_learning`.

#### Measurement (P5)
- `v_longitudinal_learning`: Spearman correlation of task success rate over time, per task type. Target: ρ > 0.3 (weak positive trend) within 90 days of activation.
- Context relevance improvement: `v_context_block_correlation` shows institutional knowledge blocks correlating with task success at higher rate than raw prior work search.
- Token efficiency: context loading with novelty-aware deduplication uses fewer tokens per task than naive similarity loading, measured via `context_profile_json`.
- Knowledge diversity: entropy of pattern distribution per task type remains above configurable threshold.
