# Voice System

## Overview

The voice system learns the inbox owner's writing style from their sent email corpus and uses it to generate drafts that match their tone, vocabulary, and patterns. This is design decision D3: voice profiles are derived from sent mail analysis, not hand-authored.

The system has six components:

1. **Sent email corpus** (`voice.sent_emails`) -- The raw material
2. **Embeddings** (OpenAI text-embedding-3-small, 1536 dimensions) -- For vector similarity search
3. **Voice profiles** (`voice.profiles`) -- Aggregate patterns extracted from the corpus
4. **Few-shot selector** -- Retrieves relevant examples for the responder agent
5. **Edit tracker** (`voice.edit_deltas`) -- Append-only record of human corrections
6. **Feedback loop** -- Closes the loop: edit deltas correct profiles and inject correction examples into the responder prompt

## Sent Email Corpus

Bootstrapped via `npm run bootstrap-voice`, which imports sent emails from the Gmail API into `voice.sent_emails`. Each record stores:

| Column | Purpose |
|--------|---------|
| `gmail_id` | Unique Gmail message ID |
| `to_address` | Recipient (for per-recipient profiling) |
| `subject` | Subject line (for topic matching) |
| `body` | Full email body (unlike inbox emails, voice emails store the body) |
| `word_count` | For length analysis |
| `sent_at` | Timestamp |
| `is_reply` | Whether this is a reply vs. new thread |
| `embedding` | vector(1536) -- OpenAI text-embedding-3-small |
| `recipient_cluster` | Clustering result (future use) |
| `topic_cluster` | Clustering result (future use) |

Note the asymmetry: `inbox.emails` never stores the body (D1), but `voice.sent_emails` does store the body because it is the owner's own sent mail needed for voice learning.

## Embedding Generation

Embeddings are generated by `src/voice/embeddings.js` using OpenAI's text-embedding-3-small model (1536 dimensions). Requires `OPENAI_API_KEY` in the environment.

**Input text format:** Subject + recipient + body concatenated, truncated to 8000 characters.

**Batch processing:** `generateEmbeddings()` processes up to 50 emails per run in batches of 20 (OpenAI batch limit).

**Single query:** `embedText(text)` generates an embedding for a query string (used by G3 tone matching and few-shot vector search).

**pgvector index:**

```sql
CREATE INDEX idx_sent_emails_embedding ON voice.sent_emails
  USING ivfflat (embedding vector_cosine_ops) WITH (lists = 50);
```

IVFFlat with 50 lists provides approximate nearest neighbor search. Cosine distance (`<=>` operator) is used because it normalizes for text length, making short and long emails comparable.

Source: `src/voice/embeddings.js`

## Voice Profiles

Aggregate writing patterns stored in `voice.profiles` at three scopes:

| Scope | Key | Description |
|-------|-----|-------------|
| `global` | NULL | Overall writing patterns across all recipients |
| `recipient` | email address | Per-recipient patterns (built when >= 3 emails to that recipient) |
| `topic` | topic name | Per-topic patterns (future use) |

### Profile Fields

| Field | How Computed |
|-------|-------------|
| `greetings` | Top 5 first-line patterns (e.g., "Hi", "Hey", "Hello") sorted by frequency |
| `closings` | Top 5 closing patterns (e.g., "Best", "Thanks", "Cheers") from last 3 lines |
| `vocabulary` | Top 50 words (4+ chars, excluding stop words) by frequency |
| `tone_markers` | `{ formalRatio: <0-1> }` -- ratio of emails containing formal markers (Dear, Regards, Sincerely, etc.) |
| `avg_length` | Average word count across analyzed emails |
| `formality_score` | 0.0-1.0 based on formal marker frequency |
| `sample_count` | Number of emails analyzed |

### Profile Resolution

When the responder needs a voice profile for a recipient:

1. Look for a `recipient` profile matching the email address
2. Fall back to the `global` profile
3. Return null if no profiles exist (prompt instructs: "Write a professional, friendly response")

Source: `src/voice/profile-builder.js`

## Few-Shot Selector

The few-shot selector (`src/voice/few-shot-selector.js`) retrieves 3-5 relevant sent emails to include as examples in the responder's prompt. Three strategies are used in priority order:

### Strategy 1: Same Recipient (60% of limit)

Highest priority. Emails to the same person capture the specific relationship tone (formal with investors, casual with teammates).

```sql
SELECT * FROM voice.sent_emails
WHERE to_address = $1
ORDER BY sent_at DESC
LIMIT 3;  -- ceil(5 * 0.6)
```

### Strategy 2: Subject Similarity (remaining slots)

Uses pg_trgm trigram similarity to find emails with similar subjects (topic matching).

```sql
SELECT *, similarity(subject, $1) AS sim
FROM voice.sent_emails
WHERE similarity(subject, $1) > 0.1
ORDER BY similarity(subject, $1) DESC
LIMIT $remaining;
```

### Strategy 3: Vector Similarity (remaining slots)

Uses pgvector cosine distance to find semantically similar emails. The query text is the subject + body of the inbound email.

```sql
SELECT *, embedding <=> $query_embedding::vector AS distance
FROM voice.sent_emails
WHERE embedding IS NOT NULL
ORDER BY embedding <=> $query_embedding::vector
LIMIT $remaining;
```

Results are deduplicated by ID across all three strategies.

Source: `src/voice/few-shot-selector.js`

## Tone Matching (G3)

Gate G3 enforces a minimum tone match score of 0.80. The check is infrastructure-based (pgvector), not LLM-opinion-based:

1. Generate an embedding for the draft text
2. Find the 10 nearest sent emails by cosine distance
3. Compute the average cosine similarity (1 - distance)
4. Pass if average >= 0.80

```sql
SELECT 1 - (embedding <=> $draft_embedding::vector) AS similarity
FROM voice.sent_emails
WHERE embedding IS NOT NULL
ORDER BY embedding <=> $draft_embedding::vector
LIMIT 10;
```

The reviewer agent also provides an LLM-based tone score, but the one-way merge rule means the automated pgvector check cannot be overridden by the LLM. If the infrastructure says the tone does not match, the draft is rejected.

Source: `src/runtime/guard-check.js` (G3 section in `checkDraftGates`)

## Edit Tracker

Every time Eric edits a draft (via CLI or dashboard), the edit is recorded in `voice.edit_deltas`. This table is append-only (enforced by database triggers that prevent UPDATE and DELETE).

### Recorded Data

| Field | Description |
|-------|-------------|
| `original_body` | AI-generated draft |
| `edited_body` | Eric's version |
| `diff` | Line-level diff |
| `edit_type` | minor, tone, structure, content, major |
| `edit_magnitude` | 0.0-1.0 character-level change ratio |

### Edit Classification Logic

| Type | Condition |
|------|-----------|
| minor | < 10% of lines changed |
| structure | > 90% word overlap (same words, different arrangement) |
| tone | > 70% word overlap (vocabulary preserved, tone shifted) |
| content | 70-90% word overlap (substantive changes) |
| major | > 50% of lines changed |

### Why This Matters

Edit deltas are called "the most valuable table in the system" (D4) because they are the ground truth signal for voice learning improvement. Each edit tells the system exactly where its drafts diverge from Eric's preferred style. The edit rate metric also drives L0 exit criteria -- the system must achieve < 10% edit rate over 14 days to graduate to L1.

Source: `src/voice/edit-tracker.js`

## Feedback Loop

The edit tracker captures corrections, but without a feedback loop those corrections are write-only data. The feedback loop closes the gap: edits directly influence profile building and draft generation so the system converges toward Eric's actual preferences over time. See ADR-011 for the decision rationale.

### Closed-Loop Flow

```
Eric edits a draft
      │
      ▼
recordEditDelta()          ← edit-tracker.js: computes diff, classifies, stores
      │
      ├─ (if 5+ edits since last build) ──▶ rebuildAllProfiles()  [async, non-blocking]
      │                                            │
      │                                    analyzeEditDeltas()     ← profile-builder.js
      │                                            │
      │                                    applyDeltaCorrections() ← merges into base analysis
      │                                            │
      │                                    updated voice.profiles  ← DB write
      │
      ▼
(next draft request)
      │
      ├─ getProfile()               ← corrected profile shapes greetings, closings, formality
      ├─ getRecentEditExamples()    ← edit-tracker.js: high-magnitude original→edited pairs
      │
      ▼
Responder prompt includes:
  - VOICE PROFILE (corrected by deltas)
  - PAST CORRECTIONS section (2-3 concrete edit examples)
  - EXAMPLES (few-shot sent emails)
```

### Edit Delta Analysis (`analyzeEditDeltas`)

Reads the last 90 days of `voice.edit_deltas` and extracts four correction dimensions:

| Dimension | How Detected | Profile Effect |
|-----------|-------------|----------------|
| `greetings` | First-line greeting changed (e.g., "Dear" → "Hey") | Corrected greetings get priority ordering in `voice.profiles.greetings` |
| `closings` | Last-3-line closing changed (e.g., "Regards" → "- E") | Corrected closings get priority ordering in `voice.profiles.closings` |
| `vocabularyOverrides` | Words present in edited version but absent from original (4+ chars, appearing in 2+ corrections) | Merged into `voice.profiles.vocabulary` with boosted frequency |
| `formalityCorrection` | Formal markers (Dear, Regards, Sincerely, Kindly) added or removed | Adjusts `voice.profiles.formality_score` up or down (clamped 0-1) |

Corrections are **recency-weighted**: an edit from today contributes weight 1.0; an edit from 90 days ago contributes weight 0.5. The formula is `max(0.5, 1.0 - daysAgo/180)`.

Scoping: `analyzeEditDeltas()` accepts an optional `recipient` parameter. `buildGlobalProfile()` calls it with no recipient (all deltas); `buildRecipientProfiles()` calls it scoped to each recipient's email address.

Source: `src/voice/profile-builder.js` (`analyzeEditDeltas`, `applyDeltaCorrections`)

### Profile Corrections (`applyDeltaCorrections`)

Merges correction patterns into the base voice analysis (output of `analyzeEmails`):

- **Greetings/closings**: Corrected patterns are prepended to the list, existing non-corrected patterns follow, truncated to 5 total
- **Formality score**: Base score + correction delta, clamped to [0, 1]
- **Vocabulary**: Override words merged into the frequency map with their correction count added
- **Tone markers**: Passed through unmodified (formality shifts affect `formality_score`, not marker ratios)

If corrections are empty (no edits in the window), the base analysis is returned unchanged.

Source: `src/voice/profile-builder.js` (`applyDeltaCorrections`)

### Auto-Rebuild Trigger

After every call to `recordEditDelta()`, the edit tracker checks `shouldRebuild()`. If 5 or more edits have accumulated since the last profile build (compared against `voice.profiles.last_updated`), it triggers `rebuildAllProfiles()` asynchronously (fire-and-forget, does not block the edit response).

The threshold of 5 edits balances responsiveness against rebuild cost. Each rebuild runs `analyzeEditDeltas()` + `applyDeltaCorrections()` for the global profile and every recipient profile with 3+ sent emails.

### Manual Rebuild

- **API**: `POST /api/voice/rebuild` -- returns `{ ok, profilesRebuilt, deltasAnalyzed, elapsedMs, globalProfile }`
- **CLI**: `voice rebuild` -- prints rebuild stats to the console

Source: `src/voice/profile-builder.js` (`rebuildAllProfiles`, `shouldRebuild`), `src/api.js`, `src/cli/commands/voice.js`

### PAST CORRECTIONS Prompt Injection

The executor-responder agent includes a `PAST CORRECTIONS` section in both email and Slack prompts, positioned between the voice profile and the few-shot examples. This gives the LLM concrete examples of Eric's corrections to previous AI drafts.

`getRecentEditExamples()` in `edit-tracker.js` selects 2-3 high-magnitude edit pairs (original snippet → edited snippet). Selection criteria:

- Minimum `edit_magnitude >= 0.1` (filters out trivial whitespace/punctuation edits)
- Ordered by `edit_magnitude DESC, created_at DESC` (most instructive first)
- Recipient-scoped first; falls back to global if no recipient-specific edits exist
- **90-day recency window**: Only deltas from the last 90 days are used. Edit deltas are human corrections by definition, so unlike `few-shot-selector.js` (which uses `AI_PIPELINE_CUTOFF` to filter sent emails), every edit delta is a valid training signal regardless of when it was created.

Prompt format in the responder:

```
PAST CORRECTIONS (Eric edited these AI drafts — learn from them):
- Original: "Dear John," → Corrected: "Hey John,"
- Original: "I wanted to reach out regarding..." → Corrected: "Quick question about..."
```

Source: `src/voice/edit-tracker.js` (`getRecentEditExamples`), `src/agents/executor-responder.js`
